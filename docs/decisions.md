| Section identity in `/etc/config/prism` IS the section name (16-hex uid), for every anonymous section type ŌĆö supersedes the parallel-`uid`-field scheme | Decision #14 fixed the rotating-cfgXXXX hazard for `subscription` by adding a `uid` UCI option alongside the still-anonymous section, then keying files / cross-references on the field instead of `s[".name"]`. The fix worked, but left two footguns: (a) `cfgXXXX` is still minted and is structurally identical to a "real" id ŌĆö nothing structurally stops a future patch from grabbing `section.section` again, and the rule against it was convention-only; (b) the same rotation hazard existed for every other anonymous type, most acutely for `rule` Ōåö `condition`, where `condition.rule = <rule's cfgXXXX>` is rotation-fragile in exactly the way subscription files were (deleting one rule shifts every other rule's cfgXXXX, the conditions then point at stale ids and either orphan or attach to the wrong rule). Decision: make every anonymous prism section a NAMED section whose name is a 16-hex random uid generated at create time. The section name IS the id; libuci persists it; `cfgXXXX` is never generated for prism sections anymore. Affects `subscription`, `rule`, `node`, `condition`, `customrs`, `bypass`. No `uid` UCI field on any of them ŌĆö `s[".name"]` everywhere. UI label-hiding is unchanged (`s.anonymous = true` on `form.GridSection` is a display flag and works identically on named sections). One shared JS module `view.prism.uid.generate()` (16 hex from `crypto.getRandomValues`, `Math.random` fallback) is required from `nodes.js` / `basic.js` / `routing.js` and called from every `handleAdd` override + the condition allocator (`routing.js parse()`'s `uci.add('prism', 'condition', uid.generate())`); the inlined `generateSubUid` duplications from #14 are removed. The Lua rpcd handler drops `ensure_sub_uid` and replaces it with `stabilise_section_name` ŌĆö a `c:rename` backstop that promotes any post-install hand-rolled `^cfg` section (e.g. someone running `uci add prism subscription` over SSH) to a stable uid on the first `sync_one`. The install-time migration `93-prism-migrate-named-sections` (replacing 92) handles the schema flip in a single uci-defaults pass: (1) collect every `^cfg`-named section across the 6 affected types; (2) for `subscription`, adopt any existing `uid` field from a 92-already-run router as the new section name so file paths and `urltest_regex_sources` entries already rewritten by 92 stay valid without further work; (3) rewrite `condition.rule` values from old cfg ŌåÆ new uid, using a captured rule-rename map, BEFORE any rename so the stored value still matches; (4) rewrite `node.urltest_regex_sources` entries the same way (subscription-rename map); (5) rename `/etc/prism/nodes/<old>.json ŌåÆ <new>.json` for subscriptions only (no-op on 92-migrated routers ŌĆö file is already at `<uid>.json`); (6) drop the now-redundant `subscription.<sid>.uid` field; (7) `uci rename` every section; (8) single `uci commit prism`. Idempotent (uid-shaped names are skipped) and safe to re-run. Trade-offs called out: `condition` sections get a uid name even though they're churn-heavy (recreated on every rule modal save) ŌĆö uniformity wins, and burning a handful of uids from 2ŌüČŌü┤ on every rule save is a non-concern; raw `/etc/config/prism` reads less prettily over SSH (every section now has a hex name), but `option name` carries the human label anyway. The previous decision #14 is the stepping stone this builds on ŌĆö its migration covered subscription files in production, and 93 adopts whatever uid 92 already assigned. Replaces #14 as the canonical reference. |
| Snapshot versions derived from the last `v*` tag (`_git<N>`), not from a forward-looking `PKG_VERSION` in the Makefile | The original scheme (decision: *CI: tag-driven releases; rolling snapshot pre-release; `PKG_VERSION` = next planned release*) used `<PKG_VERSION>_pre<N>-r<R>` where `PKG_VERSION` was the *next* release being worked toward and `N` was commits since the last `v*` tag. Conceptually clean ŌĆö `_pre` semantically reads as "leading up to the next release" ŌĆö but it required `PKG_VERSION` to be a correct *prediction*. When a planned `0.6.0` got pre-empted by a `0.5.1` hotfix, the dance was: edit Makefile `0.6.0 ŌåÆ 0.5.1`, tag and push `v0.5.1`, edit Makefile `0.5.1 ŌåÆ 0.6.0`. Two of those three edits were strictly unnecessary ŌĆö `release.yml` reads the version from the tag (`PRISM_VERSION` env, set from `${GITHUB_REF_NAME}`) and passes it to `package.sh`, which prefers it over the Makefile (`package.sh` `if [ -n "${PRISM_VERSION:-}" ]; then PKG_VERSION="$PRISM_VERSION"`). But the third edit (resetting `PKG_VERSION` back to the next planned after the hotfix) was real: without it, post-`v0.5.1` snapshots would have been `0.5.1_pre<N>` (= "pre-release leading up to 0.5.1"), implying the snapshots were *older* than the release they were actually built after ŌĆö wrong direction in APK ordering. The root issue was that `PKG_VERSION` was a guess that the Makefile had to keep in sync with the release timeline. Switched the snapshot path to derive everything from the most recent `v*` tag: `<last-tag>_git<N>-r<PKG_RELEASE>` where N = commits since that tag (`git rev-list --count "${LAST_TAG}..HEAD"`). Snapshots now describe what they actually are ŌĆö "N commits past v0.5.1" ŌĆö and the Makefile needs no editing across a release. `_git<N>` chosen over `_p<N>` because Alpine's aports convention reserves `_p` for "upstream patch level" (e.g. `openssh-9.6_p1` mirrors upstream's tarball name) while `_git` is the conventional suffix for downstream VCS snapshots taken after a release ŌĆö the latter is what these are. APK suffix ordering (alphabetical with `_alpha < _beta < _pre < _rc < (none) < _cvs < _svn < _git < _hg < _p`) puts `<tag>` < `<tag>_git<N>` < `<next-tag>`, so `apk upgrade` still picks newer snapshots and won't downgrade past a tagged release. **Bootstrap fallback retained:** if no `v*` tag exists in the repository, `package.sh` falls back to the old `<PKG_VERSION>_pre<N>` shape so the very first run (before `v0.1.0` exists) still produces a valid sortable version. **N=0 special case:** when HEAD sits exactly on the tag (local build at the tag, no `PRISM_VERSION` set), the suffix is dropped ŌĆö the local build version matches the tagged release version exactly. **PKG_VERSION's surviving role:** it's still consulted by the OpenWrt SDK build path (which calls `BuildPackage` and reads it verbatim) and by the bootstrap fallback above; the standalone `package.sh` builder ignores it once any `v*` tag exists. Keep it set to a sensible value (typically the next planned release) for SDK-build label correctness, but releases no longer depend on it. Release procedure shrinks from two steps (tag + push, then Makefile bump) to one (tag + push). `CLAUDE.md` Packaging section and `CONTRIBUTING.md` Versioning/Releasing sections rewritten to match; `snapshot.yml` header comment updated. |
| Subscription node files keyed by a stable `uid`, not the anonymous cfgXXXX section name | Subscription sections in `/etc/config/prism` are anonymous, and libuci does not persist anonymous section names to disk ŌĆö `config subscription` is written without a name, and the `cfgXXXX` id is re-derived from a package-wide section counter (`cfg<2-hex-index><4-hex-hash-of-type>`) on every parse. The hash component depends on the section type alone, so two subscriptions only differ in the index byte, which means **anything that shifts the position of subscription sections within the file rotates every subscription's cfgXXXX id** ŌĆö deleting a manual node that sat ahead of the block, dragging a node across the block in a GridSection sortable, hand-editing UCI from the CLI. After a rotation, each row's UCI-stored fields (`name`, `node_count`, `last_sync`, `url`) stay with the row (they're per-section content), but `/etc/prism/nodes/<old_cfgXXXX>.json` belongs to whichever row now happens to hold that cfg id ŌĆö observed symptom is "subscription 1's servers show up under sub 2, sub 2's under sub 3" with the displayed Nodes count still matching the row's last sync. `urltest_regex_sources` stored cfgXXXX section names too, so a regex group's source allowlist was equally rotation-fragile. Fix: every subscription section carries a `uid` UCI option (16 hex chars from `/dev/urandom` / `getRandomValues`), and every cross-session reference to "which subscription" uses uid instead of section name. Files become `/etc/prism/nodes/<uid>.json`; `build-config`'s `_source` field on imported nodes is the uid; `list_outbounds` emits `n.subscription = uid`; the JS `subName` / `subOrder` maps in `nodes.js`, `routing.js`, `status.js`, `basic.js` are keyed by uid; the `urltest_regex_sources` dropdown stores uids. The section name stays anonymous (the UI remains unchanged ŌĆö `s.anonymous = true`), and stays volatile ŌĆö it is just no longer load-bearing. Three places allocate a uid: the GridSection `handleAdd` overrides in `nodes.js` and `basic.js` (synchronous `uci.set` on the freshly-added section, before the modal opens, so the uid is staged in the same panel save as the row itself); `sync_one` in `luci.prism` as a defensive backstop for hand-rolled sections via `ensure_sub_uid`; and the install-time `92-prism-migrate-subscription-uid` uci-defaults script, which assigns a uid to every pre-uid section, renames matching `<cfgXXXX>.json` ŌåÆ `<uid>.json` in lockstep, and rewrites `node.*.urltest_regex_sources` entries that referenced the old cfgXXXX. Migration assumes current state is *aligned* (cfg ids match files on disk): a router that has already rotated without re-syncing will have its uid assigned fine but the orphan files dropped on the next sync ŌĆö the user re-syncs and everything repopulates under the new uid, which was already the only recovery path before. Three things deliberately kept simple: (a) one uid per section, no rotation/rekey ever (collision space is 2ŌüČŌü┤, fine for a per-router list of Ōēż a few subscriptions); (b) the migration runs once and is removed by uci-defaults ŌĆö subsequent installs find every section already has a uid and no-op; (c) the JS-side uid generator is inlined in both `nodes.js` and `basic.js` rather than promoted to `lib/` ŌĆö eight lines ├Ś two callers is below the threshold for a shared module. |
| Live traffic digest on Status: bytes/sec, connections, session totals, sing-box memory ŌĆö all via clash `/connections` only | The Status page already pulled `/proxies` for the Groups panel; the gap was "is anything actually flowing?", which the running badge alone can't answer (a wedged tunnel often looks running). The clash API exposes three nominally-relevant endpoints ŌĆö `/traffic`, `/memory`, `/connections` ŌĆö but `/traffic` and `/memory` are documented as WebSocket-streaming in upstream clash and sing-box's HTTP-GET behaviour on them is not something we wanted to depend on without on-hardware verification. **`/connections` is a plain HTTP GET** that returns `{ downloadTotal, uploadTotal, memory, connections: [...] }` in a single response, so one fetch covers all four signals we wanted: totals (the response itself), throughput (diff the totals across ticks for bytes/sec), memory (`memory` field ŌĆö defensively typed, hidden when absent so a future sing-box that drops the field doesn't render a "memory: 0 B" row), and open-connection count (`#connections`). We piggy-back on the existing `active-watch` daemon ŌĆö same 10 s tick, same gating on `clash_api_enabled`, same atomic-rename snapshot pattern ŌĆö writing a separate file `/var/etc/prism/.clash-stats.json`. The two snapshots are *kept separate* deliberately: `.active-nodes.json` is read by both daemon (for diff) and rpcd (for the Groups panel), while `.clash-stats.json` is write-only from the daemon's side, and merging them would force `get_active_groups` to load fields it doesn't need (and vice versa for `get_clash_stats`). **Rate semantics:** the daemon stores `prev_total_up/down/ts` across ticks and computes `bps = (new - prev) / dt`; on the first tick (no prev), on a `pid_changed` tick (sing-box restarted, counters reset to zero), or on any counter regression (defense against an unobserved restart), rate is zeroed for that tick and the baseline re-armed ŌĆö so a restart produces one `0 B/s` tick rather than a spuriously negative reading or a giant spike. **Resolution trade-off:** the dashboard ticks every 2 s but the underlying data refreshes every 10 s, so users see the same throughput value for ~5 ticks in a row; acceptable for a "is it flowing?" indicator. Sub-second granularity would require a browser-side WebSocket to `/traffic`, which is the natural follow-up if anyone needs it. **UI:** a single horizontal row above the Groups panel ŌĆö `Ōåō <rate>  Ōåæ <rate>  <N> connections  session: <totalŌåō> Ōåō / <totalŌåæ> Ōåæ  [memory: <mem>]` ŌĆö rendered as inline flex pieces, not a table (it's one reading at a moment, not a growing list). Each value span is id'd so `_updateClashStats` does an in-place `textContent` swap every 2 s without rebuilding the row (avoiding a layout flicker every poll). The whole section is hidden when (a) clash API is disabled ŌĆö RPC returns `error: "clash API disabled"`, same convention as `get_active_groups` and `test_node`; or (b) sing-box isn't currently running ŌĆö the snapshot file persists in tmpfs across a stop, so a frozen reading from a stopped service is worse than silence. Visibility coordination between `_updateStatus` (knows running state) and `_updateClashStats` (knows error state) goes through `self._running`, so each handler can hide-on-its-own-signal without contradicting the other. **What we deliberately did NOT include:** the per-connection list (the `connections: [...]` array) is parsed only for its length ŌĆö never transmitted through rpcd to the browser. A busy router can have hundreds of connections ├Ś ~20 fields each; that's a memory-budget violation on a constrained device and a layout problem on a glance-first page. If we ever want a "top talkers" view it should be a separate on-demand modal with its own RPC, not bolted onto the Status poll. ACL adds `get_clash_stats` to the `read` list. |
| Active-node visibility: Status page shows each urltest/selector's current member, and a background daemon logs every flip | Builds on the clash API plumbing decision #1: with sing-box's `/proxies` endpoint already available on loopback, the user's question "how often is `auto` actually flipping members, and is my `tolerance` too tight?" had no answer in the UI ŌĆö the Status page's `via <group>` line was a static UCI read that never reflected the live selection. Two pieces. (1) **`/usr/libexec/prism/active-watch`** ŌĆö a Lua daemon that GETs `/proxies` every 10 s, diffs each urltest/selector group's `now` against a tmpfs snapshot at `/var/etc/prism/.active-nodes.json`, and writes one `daemon.info` syslog line per change: `group 'auto': active node HK-01 -> HK-03 (142ms)` (the latency is from the new member's `history`, not the group's). Internal helpers whose tag starts with `_` (notably `_prism_test_all`) are filtered out ŌĆö they're machinery, not user-configured groups, and would otherwise spam the log. The daemon is a second procd instance declared by `/etc/init.d/prism` alongside sing-box, gated on `prism.global.clash_api_enabled=1` ŌĆö when the flag flips off, the reload re-enters `start_service`, `prism_declare_active_watch` no-ops, and procd's instance reconciliation tears the daemon down. The active-watch instance's `procd_set_param file` points at the daemon script itself, so a package upgrade that rewrites the script triggers a procd-driven restart on the next reload (without this, the old Lua interpreter stayed alive indefinitely and quietly ran pre-upgrade behaviour ŌĆö observed once during this work). Snapshot lives on tmpfs so a reboot primes it cleanly: the first successful tick after boot stashes the current state without emitting any "changed" line. Polling at 10 s is ~6├Ś oversampling for the typical 60 s urltest interval ŌĆö fast enough to catch every flip, idle cost is a single loopback HTTP round trip plus a ~20 KB JSON parse, well under the router's noise floor. The daemon never traps signals: SIGTERM from procd interrupts `os.execute("sleep ŌĆ”")` and the Lua process exits at its default disposition, which is enough ŌĆö no in-flight write to leak since the snapshot is renamed atomically. (2) **Status page Groups panel** ŌĆö a `table.cbi-section-table` between the runtime row and the log boxes, one row per group: `tag ŌåÆ now  <latency-badge>  <type ┬Ę interval ┬Ę ┬▒tolerance>`. The latency badge reuses the same colour bands as the Nodes panel (`label-success` <300 ms, `label-warning` <800 ms, `label-danger` Ōēź800 ms) but the function is copy-pasted rather than extracted into a shared module ŌĆö one helper isn't worth a new require for a single consumer. UI tuning metadata (interval / tolerance / selector default) is joined in by the RPC from UCI, not pulled from the clash response ŌĆö the API doesn't surface those fields, and putting them next to the live `now` is the whole point: the user sees `auto ŌåÆ HK-03 (142ms)  urltest ┬Ę 1m0s ┬Ę ┬▒50ms` and has the tuning right next to the evidence of churn. The panel hides entirely when the snapshot is empty or clash API is off, so a feature-flagged-off install collapses back to the original layout with no dangling "Groups: ŌĆö" placeholder. (3) **`get_active_groups` RPC** ŌĆö reads the daemon's snapshot rather than fetching `/proxies` itself, so the 2 s UI poll stays cheap (one file read, no HTTP) and the UI sees the same view the daemon is logging against ŌĆö a flip in the log and a flip in the panel are guaranteed to refer to the same observation. ACL adds it to the `read` list. Returns `{error: "clash API disabled"}` when the feature flag is off, matching the `test_node` convention. The deferred *active-member dropdown on Status* from decision #1 is intentionally still deferred: this entry adds read-only visibility, not a write path. (4) **Restart-event annotation** ŌĆö the daemon's log distinguishes three "current selection" contexts so a syslog grep can tell them apart: `starting` (cold prime, first successful poll), `after sing-box restart` (procd-reported pid changed since last tick ŌĆö uses ubus `service list` to read `instances.singbox.pid`), and `resuming` (fetch streak recovered after >=1 missed poll, sing-box pid unchanged ŌĆö the network had a hiccup, not a restart). A failure-streak heartbeat fires one `warning` line at ~60 s of consecutive fetch failures and one `info` recovery line when the streak breaks, so silent staleness can't be mistaken for "everything is calm". Group lifecycle is logged too: `now tracked` on appearance, `no longer present` on removal ŌĆö the user troubleshooting "why did my auto group stop working" finds the deletion event in the same log they use for flips. (5) **Singbox procd instance must be named** ŌĆö the prism service originally ran sing-box as procd's auto-named first instance; with `active_watch` added, the substring grep for `"running":true` in `service_running()` (rpcd handler) and the watchdog cron now matched *either* instance, so a dead sing-box plus an alive watcher read as healthy. **This is the LAN-blackhole bug** the watchdog exists to recover from, defeated by its own sibling instance. Fix: `prism_declare_instance` calls `procd_open_instance singbox`, and both callers walk the parsed JSON to `instances.singbox.running` specifically (`luci.jsonc` in Lua, `jshn` in the shell watchdog). Critically, **ubus's `instance` parameter on `service list` is accepted-but-ignored on at least some procd builds** (observed on a live OpenWrt 25.12 router during this work: asking for `instance:singbox` returned every instance), so the fix cannot rely on the filter to scope the reply ŌĆö it must parse the full list and walk the path. Any future code that needs "is sing-box running?" must use the same pattern; falling back to grep, even with the instance filter set, will silently reintroduce the LAN-blackhole bug. Renaming the instance forces a one-time sing-box restart on upgrade as procd swaps the old auto-named instance for the new named one. The daemon also tracks sing-box's pid via `singbox_pid()` and treats a change between successful polls as the authoritative restart signal ŌĆö required because a sub-10 s restart (the common case for a config Save & Apply) doesn't fail any 10 s poll and the fail-streak signal alone misses it. |

| Status page splits the persistent Enable toggle from transient Stop/Restart; pause lives on a tmpfs marker | The previous model (homeproxy-style, decision #46) collapsed "autostart at boot" and "is it running right now" onto the single UCI `enabled` flag, with Stop/Start flipping it as a side effect. That conflated two genuinely different user intents: (a) "install Prism but keep it dormant ŌĆö I'll turn it on later" (a persistent configuration choice) and (b) "stop sing-box for a minute to confirm an issue isn't its fault" (a transient diagnostic action). The collapsed model also surfaced a first-install dead-end ŌĆö the Status page showed only Stop + Restart, but `restart` did not flip `enabled` back to 1, so a freshly-installed Prism (default `enabled=0`) could not be started from the browser at all. New model: two independent controls on the Status page. (1) **Enable toggle** at the top ŌĆö a checkbox writing UCI `global.enabled` via a new `set_enabled(enabled)` RPC. Off = installed-but-dormant: `start_service` early-returns, watchdog exits, the runtime row is hidden entirely (`display:none`, contents still rendered so polling and re-enable don't need a full re-render). (2) **Stop / Start / Restart** as runtime-only buttons that never touch `enabled`. Stop writes a tmpfs pause marker (`/var/run/prism.paused`) before invoking `/etc/init.d/prism stop`; `start_service` and the cron watchdog both honour the marker and refuse to start a paused service. Tmpfs is the natural reset ŌĆö a reboot clears the marker, so an enabled-but-paused service resumes automatically and a user who forgets to click Start after testing isn't permanently broken. Start and Restart both remove the marker before execing (otherwise `start_service` would refuse). | `+lua` and `+libuci-lua` declared explicitly in `LUCI_DEPENDS` | Real-world failure: a user uninstalled `passwall2` and prism stopped starting with `env: can't execute 'lua': No such file or directory`, then after installing `lua` with `config: uci binding not available`. apk's auto-removal of "now-unused" packages orphaned both because passwall2 was the only installed package keeping them pinned in apk's needed set. On OpenWrt 25.12 apk, neither `luci-base` nor `luci-lib-jsonc` pulls in `/usr/bin/lua` or `libuci-lua` transitively ŌĆö `luci-lib-jsonc` only links against `liblua5.1.5` (the shared library, not the interpreter binary) and `libuci-lua` is an independent Lua binding package not depended on by luci-base. prism however needs the interpreter for `#!/usr/bin/env lua` on every helper (`build-config`, `fetch-catalog`, `luci.prism`) and the `uci` Lua module for `require "uci"` in `build-config`. Declaring both explicitly makes apk treat them as direct deps of `luci-app-prism`, so removing an unrelated package can never orphan them. Side note from the same incident: after `apk add lua libuci-lua` restored the bins, rpcd's in-memory object registry was still stale ŌĆö `ubus call luci.prism get_status` returned "Not found" until `service rpcd restart` re-scanned `/usr/libexec/rpcd/`. rpcd does not re-scan when handlers' dependencies change at runtime, only at its own startup. No code change needed for that ŌĆö the postinst already does `service rpcd reload` on install, and the dep declarations now prevent the orphan path in the first place. |
| Basic vs Advanced UI modes, gated only at the builder | `prism.global.mode` is `basic` or `advanced` (default `basic` on fresh installs, fallback `advanced` when unset so today's configs upgrade non-destructively). The flag decides which tabs the host view renders (Status + Basic, or Status + Nodes + Routing + Settings) and which UCI sections `build-config` reads. Manual nodes (`prism.node`), custom rules (`prism.rule`) and `/etc/prism/extra.json` are *kept on disk* in Basic mode; the builder simply ignores them. That makes the switch fully non-destructive in both directions ŌĆö AdvancedŌåÆBasicŌåÆAdvanced restores everything. Basic synthesises its config from `prism.basic.{server,routing,bypass_country,ports}` + subscriptions: N=1 selected ŌåÆ that node is the final outbound, N>1 ŌåÆ a urltest tagged `basic-auto` (displayed as "Auto" on Status); country bypass emits sagernet `geoip-<cc>` + `geosite-<cc>` rule-sets matched into one OR-ed direct rule; common-ports is a fixed list of 16 ports (SSH/mail/DNS/web/DoT/IMAP/POP3/XMPP-fallback/alt-web) emitted as a single `port` matcher. Basic also forces `firewall.sh` to drop the OUTPUT-chain rules (router-originated traffic stays untunneled) regardless of the saved `inbounds.tproxy_self` value, and hides the "Subscriptions/Active rules" + sing-box-version footer rows on Status. DNS leak edge case noted in code: with fakeip off, bypassed-country domain lookups still exit via `remote` because `prepare_rules` returns `{}` and no per-rule DNS mirrors are synthesised ŌĆö the resolution is over DoT so content is encrypted, classed as wart not leak. Toggle lives in two places (right-aligned chip in the tab bar + a friendlier link at the bottom of the Basic panel) per the design discussion; AdvancedŌåÆBasic is gated by a one-step modal so the change is never silent, BasicŌåÆAdvanced flips silently. |
| Mode naming: basic / advanced, not basic / expert | Initially shipped on-branch as "basic vs expert" because the Settings tab already had an in-tab "Show advanced settings" reveal ŌĆö using "Advanced" as the mode name would have collided ("the Advanced settings of Advanced mode"). With Basic mode absorbing the on-ramp audience, the in-Settings reveal was relabelled "Show all fields" (still useful ŌĆö there are 12 fields + 2 whole sections + the extra.json editor behind it, more than we'd want to flatten by default). That freed the word "Advanced", which pairs more naturally with "Basic" and reads less gatekeepy than "Expert". Rename is mechanical: UCI value, `set_mode` validation, build-config gate, JS string comparisons, UI labels, code comments. Unreleased branch so no migration shim; the unknown-value fallback in `read_mode` / `get_status` is "advanced", which auto-coerces any lingering `mode=expert` from earlier snapshot installs into the new default. |
| `tls.insecure` from subscription nodes is preserved, not stripped | The subscription sanitiser in `build-config` originally stripped `tls.insecure` from imported nodes (it wasn't in `SUB_TLS_KEYS`) on the theory that a hostile sub publisher could silently disable cert verification. Real-world consequence: a large class of providers ŌĆö particularly Chinese-market ones ŌĆö set `skip-cert-verify: true` per proxy because they use decoy/fronted SNIs (e.g. `down-30005.sz.gfp.tencent-cloud.com`, `*.oss-cn-shanghai.aliyuncs.com`) that the real backend's certificate doesn't cover. Prism's strict mode turned every such node into `tls: failed to verify certificate: x509: certificate is not valid for any names, but wanted to match ŌĆ”` and the entire sub was unusable, while Clash / Clash Meta / sing-box's own remote-config loader / every phone client honoured the flag and connected fine. Investigation case: a SSRDOG-style Clash YAML with 41 anytls nodes, all unreachable. The narrow attack the sanitiser was defending against (sub URL takeover ŌåÆ flipping insecure on nodes the user already trusted with credentials and routing) does not outweigh making Prism strictly less compatible than every other client. Decision: add `insecure` to `SUB_TLS_KEYS` so the publisher-declared per-node value flows through verbatim. The rest of the strip list (`detour`, `ssh.private_key_path`, transport overrides, unlisted top-level keys, anything not in each protocol's `SUB_TYPE_KEYS` whitelist) stays ŌĆö those are still defended. `disable_sni` was considered for the same treatment but left stripped: legitimate providers don't need it (they set `sni` to the decoy and accept the cert mismatch via `insecure`), so its only practical use is a hostile publisher deliberately suppressing SNI to ease MITM. |
| Selector node type removed; the only group type is now `urltest` | Decision #28 added the `selector` node type as a manual-pick group sharing `urltest_outbounds` plus a `selector_default` field. In practice it was half-implemented: the editor stored `selector_default` but Prism shipped no runtime switcher (the deferred Status dropdown from decisions #17 / #27 never landed), so a selector group behaved as a static pin to one node ŌĆö a job a routing rule pointing directly at the node, or the Default-server picker, already does without the extra indirection. urltest stays useful because auto-rotation by latency isn't replicable any other way; selector was not earning its UI surface. Removal scope: the `selector` entry is dropped from the Type dropdown (`nodes.js`); `selector_default` and its dynamic `ui.Dropdown`-via-renderWidget plumbing (the `setTimeout(0)` clone-map lookup pattern from decision #28) are gone; the modal tab reverts from "Group" to "URLTest"; basic.js / status.js / routing.js comments drop their selector references and the Status _renderGroupMeta selector branch is removed. `build-config`'s `build_node_from_uci` no longer builds a selector outbound ŌĆö a legacy UCI `type=selector` section is converted in-memory to `type=urltest` (member list preserved, `selector_default` dropped) so an upgraded install with stored selectors keeps routing without breaking sing-box. Step 3's group expansion and step 4's empty-group skip both narrow back to `urltest` only. **The `_prism_test_all` hidden helper** stays a `type="selector"` outbound: it's a sing-box-level construct that holds every concrete-endpoint tag for clash-API batch probing, and selector is exactly the right sing-box type for "group that never auto-probes" ŌĆö the user-facing removal is about the *node creation surface*, not sing-box's outbound type itself. `_prism_test_all` is filtered from `active-watch`'s group snapshot by the existing underscore-prefix rule plus a defensive `p.type == "URLTest"` narrowing, and from the Basic node picker by `GROUP_TYPES = { urltest: true }` (clash-format subscriptions can't smuggle selectors through anyway ŌĆö `SUB_ALLOWED_TYPES` only admits concrete protocol types, so no remote selector ever reaches Prism). `luci.prism` gets a `migrate_legacy_selectors` helper invoked once per RPC `call` dispatch: it rewrites any UCI `node` section with `type=selector` to `type=urltest`, deletes the `selector_default` option, commits, and logs `migrated legacy selector node(s) to urltest`. The migration is a no-op on clean installs and fires at most once per upgraded router (after the first RPC, the UCI state is canonical and subsequent calls find nothing to migrate). The in-memory fallback in `build-config` is the safety net for the narrow window where build-config runs before any rpcd RPC has ŌĆö e.g. boot path, hand-edit via SSH between an upgrade and the first UI visit. No standalone uci-defaults migration script: the dispatch-time helper covers every realistic upgrade path and avoids a one-shot pass that needs its own idempotency guard. ACL is unchanged (the migration runs as part of the existing handler, no new method). `get_active_groups` drops the `s.type == "selector"` UCI scan branch, the `entry.type == "Selector"` snapshot translation, and the `selector_default` lookup. The deferred Status-tab runtime switcher mentioned in #17 / #27 stays deferred ŌĆö removing selector doesn't preclude re-adding it later if the indirection-with-runtime-flip use case proves valuable; the rebuild would be a write-path RPC (`/proxies/<tag>` PUT) plus a single `ui.Dropdown` in `_renderGroups`, not a re-introduction of the editor UI. |
