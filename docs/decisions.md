# Prism — Decisions & Context

Running log of design decisions and solved problems. Update this file
whenever a non-obvious choice is made or a tricky problem is resolved, so
future work has the rationale available.

This is the long-form record; for actionable rules and conventions, see
`CLAUDE.md`. Before reversing a design choice or revisiting a settled
question, check here first.

| Decision / Problem | Rationale |
|---|---|
| Status page splits the persistent Enable toggle from transient Stop/Restart; pause lives on a tmpfs marker | The previous model (homeproxy-style, decision #46) collapsed "autostart at boot" and "is it running right now" onto the single UCI `enabled` flag, with Stop/Start flipping it as a side effect. That conflated two genuinely different user intents: (a) "install Prism but keep it dormant — I'll turn it on later" (a persistent configuration choice) and (b) "stop sing-box for a minute to confirm an issue isn't its fault" (a transient diagnostic action). The collapsed model also surfaced a first-install dead-end — the Status page showed only Stop + Restart, but `restart` did not flip `enabled` back to 1, so a freshly-installed Prism (default `enabled=0`) could not be started from the browser at all. New model: two independent controls on the Status page. (1) **Enable toggle** at the top — a checkbox writing UCI `global.enabled` via a new `set_enabled(enabled)` RPC. Off = installed-but-dormant: `start_service` early-returns, watchdog exits, the runtime row is hidden entirely (`display:none`, contents still rendered so polling and re-enable don't need a full re-render). (2) **Stop / Start / Restart** as runtime-only buttons that never touch `enabled`. Stop writes a tmpfs pause marker (`/var/run/prism.paused`) before invoking `/etc/init.d/prism stop`; `start_service` and the cron watchdog both honour the marker and refuse to start a paused service. Tmpfs is the natural reset — a reboot clears the marker, so an enabled-but-paused service resumes automatically and a user who forgets to click Start after testing isn't permanently broken. Start and Restart both remove the marker before execing (otherwise `start_service` would refuse). Three runtime states show distinct badges + button sets: **Running** (`● Running` + Stop + Restart), **Paused** (`○ Paused` + Start + Restart + "Will resume on next reboot" note), **Stopped** (`● Stopped` red + Start + Restart — typically a crash that exhausted procd's respawn cap before the */5 watchdog re-armed). `get_status` returns the new `paused` field so the frontend can distinguish paused from crashed. The Autostart read-out is dropped from the footer (the Enable toggle is now its visible control). ACL gains `set_enabled` on the write list. Partially reverses decision #46: that decision was right to make `enabled` the boot gate, but wrong to make Stop write it; this re-splits the runtime concern off `enabled` without re-introducing the persistent manual/autostart flag pair the old `manual` field was. Supersedes the just-shipped *Status page gains a Start button when sing-box is stopped* (the Start button is still there, but its semantics are now "resume from pause / recover from crash", not "set enabled=1"). |
| UCI ↔ UI parity: every UCI option is editable from the UI; every user-editable UI field persists to UCI | Two-way invariant. (1) **UCI → UI**: every option in `/etc/config/prism` must have a control in some tab — otherwise the only way to change it is SSH + `uci set`, which defeats the point of shipping a web UI. Surfaced by the first-install Start-button gap: `global.enabled` ships at `0` and the Status page only exposed Stop + Restart; the `restart` RPC at `root/usr/libexec/rpcd/luci.prism` runs `/etc/init.d/prism restart` but does **not** set `enabled=1`, and `start_service` early-returns when `enabled=0`, so a freshly-installed Prism could not be turned on from the browser at all. Fix lives in the same change as this entry — see *Status page gains a Start button when sing-box is stopped*. (2) **UI → UCI**: anything the user can change in the UI must round-trip to UCI (or to another tracked persistent store like `/etc/prism/extra.json` or `nodes/*.json`) — no in-memory-only toggles, no UI state that vanishes on reload. The right home for a setting is the UCI section that matches its scope (`global`, `inbounds`, `dns`, `routing`, …) and a `form.Map` row in the matching tab. Runtime-only controls (Stop/Start/Restart on Status) are exempt because they're actions, not settings — but if such an action mutates a UCI flag as a side effect (Stop sets `enabled=0`), the inverse action must also exist somewhere in the UI (a Start path that sets `enabled=1`). Exceptions go through this log with a rationale. |
| Status page gains a Start button when sing-box is stopped | The 2026-05-22 layout reasoned that "Restart on a stopped service just starts it, so no separate Start button — Stop and Restart cover every transition." That premise is wrong: the `restart` RPC at `root/usr/libexec/rpcd/luci.prism` only execs `/etc/init.d/prism restart`, while `enabled=1` is only set by the separate `start` RPC. Combined with `start_service`'s `[ "$enabled" = "1" ] || return 1` gate (the single-source-of-truth rule from *Service start path rebuilt on homeproxy's model*), Restart on a stopped+disabled instance is a no-op. On a fresh install `/etc/config/prism` ships `option enabled '0'`, so a new user had no in-UI path to bring Prism up — the violation of the UI ↔ UCI parity invariant that motivated the entry above. Fix: `status.js` adds a `callStart` RPC binding to the already-implemented `start` method (which flips `enabled=1` and execs `start`), a `handleStart` handler, and renders the action buttons dynamically — Start alone when stopped, Stop + Restart when running. The button container is regenerated by `_updateStatus` on every poll tick so the toolbar follows state changes (poll noticing a crash, a successful Stop, etc.) without needing a separate re-render path. The stale comment block above the status row is updated to describe the new three-state toolbar. |
| Orphan `/etc/prism/nodes/<sub_id>.json` files are pruned on every sync, and skipped on read | `luci.prism` had no path that removed a node file when its UCI subscription section was deleted (`form.GridSection`'s standard Remove writes UCI directly with no RPC hook). A deleted-then-recreated subscription, or a manual UCI edit that changed a section id, left a file on disk whose sub_id no longer matched any UCI section — `list_all_nodes` still loaded it, surfaced its tags in every outbound dropdown (Routing's Server picker, Default-server, urltest member multi-select), and the labelling fell back to bare-tag because `subName[ob.subscription]` was undefined — making the orphan's tags look like manual nodes and, worse, collide with valid subscription tags that share names. A new `valid_sub_ids()` helper enumerates current UCI `subscription` sections; `list_all_nodes` consults it and skips files whose sub_id isn't in the set (pure read, no delete here — surprising side-effects in a read path). The actual disk delete lives in `prune_orphan_nodes()`, called at the end of `sync_subscription` and `sync_all_subscriptions` so a user-triggered sync or the hourly cron also clears the file — bounding the on-disk lag to one cron tick. Not all-encompassing: it does **not** detect a *stale-content* file (subscription still exists in UCI but its file holds content written from a previously-configured URL — Prism has no record of past URLs, and a successful sync overwrites the file anyway). That case is operator-resolved (re-sync the subscription, or remove and re-add it). |
| Per-rule DNS detour: each routing rule can resolve its domain matchers through the same outbound the connection takes | Previously `build_routing_dns_rules` mirrored every proxied rule's domains onto a single `remote` DNS server, which itself detours through `route.final` (the Default server). So a rule routing Disney to `法国 01` had its DNS resolved through whatever the user picked as Default (e.g. `V1 SouthKorea`) — geographically inconsistent with the connection, and visibly wrong for CDNs that geolocate by resolver IP (Netflix in particular). New per-rule flag `rule.dns_via_outbound` (UCI option, default `1` — unset is treated as on so existing rules upgrade transparently) makes `build_routing_dns_rules` mirror the rule onto a dedicated `remote-<outbound-tag>` DNS server whose `detour` points at the rule's own outbound. `build_dns` emits one such server per distinct flagged outbound (deduplicated, sorted for deterministic output, same upstream address as the main `remote` so the user's resolver choice carries through). Unflagged rules, `direct`-routed rules and fakeip-mode rules keep the previous mirror targets exactly. UI surfaces a Flag option under the rule modal's outbound picker; it depends-shows only when the picked outbound is a real proxy tag — `direct` resolves via `local` and `block` produces no DNS rule, so the flag is moot for both. LuCI's `depends()` has no "not equals" form, so `addServers()` now returns the proxy tag list it added and the depends loop ORs one clause per tag. Cost: one extra DNS server per distinct outbound used by flagged rules + a first-query connection bootstrap on each (negligible — those outbounds are reused for traffic anyway). Benefit: CDN region selection follows the proxy exit instead of the default outbound. Sing-box's own DNS server tag namespace is separate from the outbound namespace and accepts arbitrary strings, so `remote-V1 SouthKorea` / `remote-法国 01` work as tag names without escaping. |
| `build_route` emits an unconditional `ip_is_private=true → direct` rule so LAN/loopback/link-local/ULA destinations bypass the proxy without requiring a user routing rule | In tproxy / tproxy_mixed mode `firewall.sh` already filters RFC1918, link-local, multicast, CGNAT and friends at the nftables prerouting stage (the `V4_RESERVED` / `V6_RESERVED` sets), so a `final_outbound` set to a proxy server still leaves LAN-to-LAN traffic on the local network. In **tun** mode there is no such firewall hook — TUN captures every IP packet — so a user who picked TUN + a proxy as their Default server would silently tunnel LAN traffic (router admin, file shares, printers) into the proxy and break LAN access; the previous `build-config` emitted nothing for the private-IP case. The fix is a single route rule injected in `build_route` after the DNS-hijack pair and before any user-defined rule: `{ ip_is_private = true, outbound = "direct" }`. The matcher (`route/rule/rule_item_ip_is_private.go`, `sing/common/network/addr.go:IsPublicAddr`) matches RFC1918 + loopback + multicast + link-local + ULA + unspecified, and intentionally *does not* match CGNAT (100.64/10) or the IPv4 fakeip default (198.18/15). Verified available since sing-box 1.11 (`option/rule.go` at v1.11.0 and v1.12.0), so it works on Prism's whole 1.12+ floor. **Fakeip safety**: sing-box demaps fakeip → FQDN at the *start* of `matchRule` (`route/route.go:434-448`), before the rule-evaluation loop; after demap `metadata.Destination.Addr.IsValid()` is false and `IPIsPrivateItem.Match` (rule_item_ip_is_private.go:30) returns false. So even with IPv6 fakeip on (whose default range `fc00::/18` sits inside ULA `fc00::/7`) fakeip routing keeps working. In tproxy mode the rule is inert and harmless. A user who needs to proxy a private destination (rare — e.g. routing to a private IP at the far end of an existing VPN) can place their rule above it; left unconditional rather than behind a UCI toggle because a toggle would add a setting nobody should need to touch. |
| Settings page reorganised; two NamedSections share the `global` UCI section (Rule-sets + Logging) | The 2026-05-23 reorg lumped log levels and rule-set fetching into one "General" section because of a constraint stated as "multiple `form.NamedSection`s binding the same UCI section name would collide on the generated DOM id." Re-reading `form.js`'s `NamedSection.renderContents` (form.js ~4271/4274 in current LuCI master): the wrapper `<div class="cbi-section">` gets `id="cbi-<config>-<section_id>"` (only when the UCI section is absent), and the inner `<div class="cbi-section-node">` gets the same id (when present). The *per-option* ids are `cbid.<config>.<section_id>.<optname>` — keyed on the section_id string, not on the JS section object. So two NamedSections both bound to `('global', 'prism')` only duplicate the wrapper-div id (HTML-invalid but functionally inert — no form code keys off it); every per-option cbid stays unique as long as no option name repeats across the two sections, which is what save/load/validate/`getUIElement` actually rely on. The constraint as previously stated was overstated. The Settings page now uses two real sections — `Rule-sets` (delivery + download_detour) and `Logging` (sing-box + Prism log levels) — both binding `global`. The Logging section is wholly Advanced: its wrapper `.cbi-section` gets the `prism-advanced` class in `_postRender` (located by walking up from its first option's input element via `firstOpt.cbid(section.section)` lookup, since `section.children` is the framework's option array), so the section title + every row hide as one unit. This replaces the previous "inject an h3 subheader before log_level" hack and drops the `_prismSubheader`/`prism-subheader` CSS plumbing. Other reorg moves: `tproxy_self` (Proxy router traffic) demoted from Advanced → Standard (no day-to-day reason to disable); `managed_dns` promoted to Advanced (default on, no standard reason to disable); `log_level` + `prism_log_level` + `ruleset_download_detour` moved to Advanced; mode-specific Network rows (tproxy / tproxy_mixed / tun groups) reordered to sit directly under `Mode` so each mode's options are visually adjacent to the selector that reveals them; `Enable IPv6` moved to the bottom of Network as the only mode-independent row. Section order on the page is now Network → DNS → Rule-sets → Logging → Raw overrides, reading top-to-bottom as the traffic flows then ending with operator/diagnostic knobs. |
| `country` condition kind removed; `country_provider` setting removed | Anyone touching routing rules already understands the rule-set catalog, so the `country=cn` shortcut — which only expanded to `<provider>/geoip-cn` plus `<provider>/geosite-cn` (when the catalog confirmed the geosite existed) — was sugar for a user who already knows what those tokens mean. Removing it deletes one condition kind, the 30-entry `COUNTRIES` autocomplete array, the `countrySuggest` plumbing on `ConditionList`, the `country_provider` dropdown in Rule-set sources, the matching UCI default, the `else  -- country` branch in `build-config:resolve_condition`, the `country_provider` / `catalog_has` keys in `prepare_rules`'s ctx, and `load_catalog()` (its only consumer was the country geosite-availability gate; `rulesets-cache.json` itself stays — `luci.prism`'s `list_rulesets` still reads it for the autocomplete catalog). A country-style match is now expressed as two `ruleset` conditions joined by OR (`sagernet/geoip-cn` OR `sagernet/geosite-cn`) — the same two rule-sets the shortcut emitted, just spelled out. No migration script: existing `kind=country` conditions become no-ops (`compile_condition` falls through to `node = nil`, and a rule whose every condition resolves to nothing is skipped); the user re-adds them as ruleset conditions on upgrade. The legacy `90-prism-migrate-rules` / `91-prism-migrate-conditions` uci-defaults still emit `country` for pre-2026-05-22 schemas, which will likewise become dead rules on a first-time upgrade from one of those very old schemas — acceptable because the audience for that path is empty in practice. |
| UI restructured to 4 flat tabs (Status / Servers / Routing / Settings); Settings uses an Advanced expander; vocabulary swap Outbound → Server | The 2026-05-22 layout (Overview / Outbounds / Routing / Settings parent with 5 sub-tabs / Diagnostics) was researched against peer sing-box clients (Hiddify, Clash Verge Rev / Mihomo Party, Nekoray, HomeProxy, GUI.for.SingBox) and against general progressive-disclosure UX literature, and replaced. Three observations drove the change: (1) every consumer-facing client renames `outbound` — sing-box internal jargon — to a friendlier noun (Clash family uses "Proxies", Nekoray uses "Servers"); the user picked **Servers**, which is now the term in tab names, the routing rule column ("Server"), the default-fallback section ("Default server") and sync notifications. The UCI schema (`node`, `subscription`), the RPC method names (`list_outbounds`) and the on-disk sing-box JSON keep the historic spelling — wire format is unchanged, only labels move. (2) the "is it working / why not?" mental flow was split across Overview and Diagnostics for no reason; they are now one **Status** tab — service controls + the *active* default server (read from `routing.final_outbound` at render time) + a fixed 20-line log tail at 2s polling + Subscriptions/Active-rules counts + a footer (`sing-box X.Y.Z · Autostart · Mode`). The full log viewer and the generated-config preview moved into on-demand modals, off the always-visible glance. (3) the 5 Settings sub-tabs became one `form.Map` on one scroll page. Basic options (log level, network mode + IPv6, managed DNS, local/remote DNS, fake-IP) are visible by default; the Advanced fields (TProxy/Mixed ports, "proxy router traffic", TUN address/MTU/stack/auto-route, DNS strategy, Fake-IP ranges) and the raw `extra.json` editor are hidden behind a single `<details>` toggle that flips a `prism-show-advanced` class on the form root — a tiny inline `<style>` block keys row visibility off that class, so the disclosure is one DOM mutation rather than a Map rebuild. Multiple `form.NamedSection`s binding the same UCI section name would collide on the generated DOM id, so each UCI section (`global`, `inbounds`, `dns`) gets exactly one NamedSection; the Basic/Advanced split is achieved by tagging individual options at construction time (an `advance()` helper pushes the option into a list) and adding the CSS class to each one's enclosing `.cbi-value` row at post-render. depends() works unchanged because all options share the same Map. The `enabled` UCI flag is intentionally **not** in Settings: Status owns runtime control via Start/Stop, so the master flag has one source of truth — a duplicate toggle would invite a sequence where the user disables in Settings, returns to Status, and sees Stopped without remembering why. The former Rule-sets sub-tab (delivery / download detour / country provider + custom rule-sets) is folded into **Routing**, since it only exists to back a rule's `ruleset`/`country` conditions — co-location avoids the tab-hop on every new rule that needs a custom rule-set. Tab redirection: stored session ids `overview`/`diagnostics`→`status`, `outbounds`→`servers`, any `settings/<sub>`→`settings`, so a Save & Apply (which reloads the page) lands on the equivalent merged tab instead of `TABS[0]`. Deliberately deferred: an inline `ui.Dropdown` group switcher on Status — it would let the user flip the active selector member from the landing page, but it is the only piece that *writes* cross-tab state and needs a focused RPC plus dropdown plumbing; for now the active group is read-only text. The first-run wizard (Hiddify pattern: paste subscription → pick mode → enable) is also deferred. |
| Selector node type added; shares `urltest_outbounds` with urltest, gets a `selector_default` field; modal tab renamed URLTest → Group | sing-box `selector` is structurally a `urltest` minus the latency probe (no `url`/`interval`/`tolerance`) plus an explicit `default` member. The member-list UCI key `urltest_outbounds` is reused for selector nodes rather than introducing a parallel `selector_outbounds` — the field is identical in shape, and a single shared `MultiValue` keeps both the form-row code and `build-config`'s prune-missing-members pass single-sourced. Selector is manual-only — regex mode would leave the default with no stable referent — so `urltest_mode` / `urltest_regex` stay urltest-exclusive; `selector_default` is selector-exclusive (a `ListValue`, optional — empty means sing-box uses the first member; the dropdown is dynamically filtered to just the section's picked `urltest_outbounds`. The widget is constructed directly as a `ui.Dropdown` inside an overridden `renderWidget` because `form.ListValue`'s default rendering uses `ui.Select` (a native `<select>`, form.js:4697) which has **no `addChoices` / `clearChoices` methods** — only `ui.Dropdown` does (form.MultiValue uses it, form.js:5236). Initial choices are seeded as `{'': '— none —'}` only; one `setTimeout(0)` later — after the modal is mounted in the DOM and the modal map's section.load chain has populated each cloned option's `.data[section_id]` — a deferred callback reads the picked tags via `pickedMembers` and calls `addChoices(picked, labels)` to fill the dropdown, then `setValue(saved)` restores the saved default if it is still a member. The deferred path is required: at the synchronous renderWidget pass during modal open, the clone's `urltest_outbounds.cfgvalue(section_id)` returns empty even though the data is set on the *outer* option — verified by re-invoking renderWidget from devtools after the modal mounts, which produces the correct dropdown. Subsequent live edits go through `urltest_outbounds.onchange` calling `clearChoices(true)` + `addChoices(...)`. **Crucial cross-map note**: `form.GridSection.renderMoreOptionsModal` (form.js:3814) builds a *separate* modal `CBIMap` with *clones* of every option via `cloneOptions` (form.js:3755). `AbstractValue.getUIElement` (form.js:1892) uses `this.map.findElement(...)` which only searches its own map's DOM. So an onchange handler that closure-captures the outer `oDefault` reaches the wrong map and gets `null` back from `getUIElement`. The handler instead resolves the sibling via `this.map.lookupOption('selector_default', section_id)` — inside the modal `this.map` is the modal's clone, so the lookup returns the modal-side clone of the option and its `getUIElement` finds the live widget. The same rule applies to `pickedMembers` reading the picked tags via `this.map.lookupOption('urltest_outbounds', section_id)`.). Step 3's group expansion is widened from `ob.type == "urltest"` to either group type so a selector's members are pulled into `referenced[]` and dangling tags pruned; the same pass also drops a stale `selector_default` whose member is no longer present. The modal tab is renamed "URLTest" → "Group" since it now covers both. `urltest` + `selector` are also moved to the top of the type dropdown — they are the two outbound-group types and the most common targets for a routing rule's `outbound`. |
| Subscriptions tab gains a per-row "View" button that opens a modal listing the synced nodes; latency deferred | The Subscriptions grid only exposed a node *count* — to see what was actually in a subscription the user had to read `/etc/prism/nodes/<id>.json` over SSH. A new per-row button calls a focused `list_subscription_nodes` RPC that reads only the requested file and returns each node projected to `{tag, type, server, server_port}` (the on-disk file also carries the multi-KB `payload` blob per node, stripped here), then renders them in a `ui.showModal` modal as a plain `table.cbi-section-table` of Name / Type / Server. An inline expand-row under each subscription was considered and rejected: `form.GridSection` has no expander hook, and adding one would mean either overriding undocumented render internals or rebuilding the list by hand — reversing the 2026-05-22 GridSection switch. A modal is the same pattern routing rules already use for editing and required no GridSection surgery. Latency was deliberately deferred to a separate decision (it needs sing-box-side measurement — a transient urltest or the clash-api `/proxies/{name}/delay` endpoint — not an rpcd-side probe that would bypass the proxy chain and tell us nothing useful). |
| Remote rule-sets omit `update_interval`; `download_detour` flagged for a future `http_client` migration | `emit_remote_ruleset` in `build-config` no longer writes `update_interval = "1d"` — sing-box already defaults an omitted `update_interval` to `1d`, so the explicit value was redundant. Separately, the `download_detour` field Prism emits to route a rule-set fetch out the WAN is **deprecated in sing-box 1.14 and removed in 1.16**; its replacement is the per-rule-set `http_client` object, which only exists from 1.14. Prism's declared sing-box floor is 1.12, so `download_detour` is currently the only spelling valid across the whole supported range (1.12–1.15) and is kept. A code comment at `emit_remote_ruleset` records that when the sing-box floor is raised to 1.14+, this must migrate to `http_client`. |
| The GridSection panels' Reset button does a global revert | `formpanel.resetGrid` did `uci.unload('prism')` + remount, which only discards *un-saved, in-memory* edits. But a GridSection edit is flushed to the backend change-staging the moment its modal is saved — the modal's Save calls `m.save()` → `uci.save()` — so by the time the panel-level Reset runs there is nothing un-saved to drop and it silently did nothing (only LuCI's global Changes-window Revert cleared staged changes). `resetGrid` now delegates to `ui.changes.revert()`, the same operation as that window. Prism only ever edits the `prism` config, so a global revert and a per-panel reset clear the same set; `formpanel.js` no longer needs the `uci` import. |
| Routing `rule` sections are created with explicit names, not anonymous | The condition model links a child `condition` to its parent `rule` by the rule's section name (`condition.rule`). LuCI gives a freshly-added *anonymous* section a temporary `newNNNN` id and renames it to the real `cfgXXXX` id when the add is flushed — but a `condition.rule` value captured as that temp id is a plain string and does not follow the rename, so every condition added in the same modal session as a brand-new rule is orphaned (the grid then shows "no conditions (skipped)", and `build-config` skips the rule). `routing.js` overrides the GridSection's `handleAdd` to create each rule as a named section (`rule_<timestamp>_<rand>`), whose id is fixed from creation through commit; `anonymous=true` still hides the name in the UI. Nothing else changes — `build-config`, `readConditions` and the `ordersave` orphan-prune already match conditions to rules by that string, which is now stable. Migration-created and pre-existing `cfgXXXX`-named rules are already stable and unaffected. |
| Routing conditions gain a per-condition NOT; the port editor becomes a comma-separated field | A `condition` may carry `invert='1'`; `compile_condition` builds the match node then sets sing-box's `invert: true` on it, negating just that one condition (e.g. "all ports except 443"). It is **per-condition** by design: sing-box also allows `invert` on a logical block or a whole rule, but exposing those would need explicit group UI plus a compiler special-case — a single-condition rule's node would otherwise carry two colliding `invert`s — for a capability a router rarely needs, and per-condition NOT combined with the OR operator already reaches `NOT(A AND B)` via De Morgan. The UI adds an `is`/`is not` selector per condition row (a `<select>`, so it lines up with the kind selector); the grid summary and the live preview both prefix an inverted kind with `NOT` (via a shared `condLabel` helper). Separately, the `port` kind's value editor is now a single text input — comma-separated ports, `lo-hi` for ranges (`80, 443, 8000-8100`) — instead of the growable `ui.DynamicList`, which suits short port tokens far better. Storage is unchanged: the comma string splits into the same `value` list that `split_ports` already turns into `port` / `port_range`, so only the editor differs. |
| `start_service` defers the first sing-box start until a default route exists | sing-box's remote rule-set initialization and urltest probes need a route out; at cold boot the WAN (DHCP/PPPoE) is not up when `/etc/rc.d/S95prism` runs `start`, so sing-box came up routeless and flooded `no route to internet` for the whole pre-WAN window (and, with an empty rule-set cache, FATAL'd outright). `start_service` now gates the first start on connectivity: if there is no IPv4 *or* IPv6 default route it logs `service: no default route yet, waiting for WAN` and returns without declaring a procd instance — nothing started, nothing to flood. The existing `interface.*.up wan` procd trigger re-runs `start_service` when the WAN comes up (one clean start, with connectivity), and the `*/5` watchdog cron is a backstop if that event is ever missed. A `pgrep sing-box` guard limits the wait to the genuine first start: when sing-box is already running the gate is skipped, so a Save during a WAN outage cannot stop a healthy instance — keeping the 2026-05-21 "don't stop a running sing-box" posture. This closes the cold-boot error flood and the empty-cache first-boot FATAL left open by the remote-rule-set switch. The deferral is fail-safe: while the WAN is down there is no internet to leak, and nothing (sing-box, firewall, DNS drop-in) is set up until there is a route. |
| Rule-sets switched from Prism-fetched local files to sing-box `remote` rule-sets | The 2026-05-22 decision to download `.srs` files onto flash as `type:local` rule-sets was reasoned from first principles, not testing, and its `fetch-rulesets` subsystem was the source of repeated bugs. `build-config` now emits `{type:"remote", tag, format, url, update_interval:"1d", download_detour}` rule-sets — sing-box downloads, caches and refreshes each one itself and hot-loads it with no process restart. The `provider_url` URL builder moved into `build-config`; `resolve_condition` maps each `ruleset`/`country` token to a tag + download URL with no disk check. | Routing rules gain AND/OR condition composition; the flat destination schema becomes child `condition` sections | The 2026-05-22 redesign deliberately did not expose sing-box `logical` rules; this revisits that on purpose. A rule's destination is now an ordered list of conditions joined by per-row AND/OR operators. The flat `rule` fields (`domain`/`domain_suffix`/`domain_keyword`/`domain_regex`/`ip_cidr`/`ruleset`/`country`/`protocol`/`port`) are replaced by child `condition` UCI sections (`rule` back-reference, `order`, `op`, `kind`, `value` list); the `rule` section keeps only `enabled`/`name`/`order`/`outbound`. `protocol` and `port` are folded in as ordinary condition kinds so they too can be OR-mixed rather than always-AND. Precedence is standard — AND binds tighter than OR — so a rule compiles to an OR of AND-runs (sum-of-products): `build-config` splits the condition list at every `op=or`, wraps each multi-condition run in a `{type:logical,mode:and}` rule and the runs in a `{type:logical,mode:or}` rule, attaching the route action to the top node. A rule with no conditions (or whose conditions all reference a rule-set not yet on disk) is skipped, not emitted as a match-everything rule — it would shadow every rule below it, and `route.final` is already the real catch-all. Each condition compiles to its own match node, so domain and ip_cidr matchers are never placed on one rule — the per-rule inline rule-set the old code used to dodge the sing-box default-rule AND-trap is gone. DNS-server derivation (`build_routing_dns_rules`) ignores the AND/OR structure: every domain-bearing condition (literal domains, geosite rule-sets) is mirrored onto the server the rule's outbound implies — an over-approximation that only picks a resolver and never affects routing, the same simplification the old code already made for protocol/port. The UI Destination tab is rewritten as a custom modal widget (`ConditionList`, extending `form.DummyValue`): a list of `[kind][value list][remove]` rows with an AND/OR select between them and a live plain-text preview of the parsed grouping; `ui.DynamicList` supplies the per-kind value editor as plain text inputs (always free-typeable, empty by default), with a native `<datalist>` offering suggestions for ruleset/country/protocol — a choices-backed combobox was rejected because it blocked free typing and pre-filled its first entry, retiring the hand-rolled `attachAutocomplete`/`SuggestList` height-capped dropdown and its `_teardown`. The widget is backed by the child sections directly — `cfgvalue` reads them, `parse` reconciles them but only when its editor DOM is connected (the modal is open), so a modal Save writes that rule's conditions, a panel-level Save is a no-op for already-saved rules, and the `parse` of every un-opened rule during a modal save is skipped. `ordersave`'s pre-hook sweeps `condition` sections orphaned by a deleted rule. A `91-prism-migrate-conditions` uci-defaults script converts existing flat rules, distributing `(OR of destinations) AND protocol AND port` into the equivalent OR-of-AND-runs so the old semantics are preserved exactly. |
